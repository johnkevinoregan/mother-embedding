## New: `Gabor_Orientation_Demo.jl` (Julia port of the Python demo)

A faithful Julia/Pluto port of `gabor_orientation_demo.py` (the Python source
is included in the repo). At one chosen scale it convolves a character with a
bank of complex Gabor filters over orientation, takes the modulus and phase at
every pixel, keeps the argmax orientation, and produces four views:

1. the argmax-orientation flow field (HSV map + needle field),
2. raw vs modulus-masked argmax (why flat regions must be masked),
3. the full modulus(θ) profile at a stroke / junction / background
   (unimodal vs bimodal vs noise floor), and
4. phase at the winning orientation → even (line) vs odd (edge).

Port notes: `scipy.fftconvolve(…, "same")` → `imfilter(…, Fill(0))`; the
EMNIST font-render fallback is dropped (lab server has EMNIST, loaded via
`LoadEMNIST`); the whole EMNIST-balanced alphabet (A–Z, lowercase subset,
digits) is selectable, and the analysis runs only for the selected character
to stay responsive. This notebook deliberately defines its **own** Gabor bank
(anisotropic, carrier along the normal, mean-subtracted) rather than reusing
`CreateGaborLifting.gabor_kernel` — the conventions differ, and a table in the
notebook documents the differences. Orientation self-test passes
(vertical → 90°, horizontal → 0°). A one-character fix along the way: the
displayed glyph was vertically flipped (a stray `reverse`); removed so it
matches the HSV/needle panels.

## Corrected: kernels are now zero-DC

`gabor_kernel` now subtracts the kernel mean before normalizing, so the complex
kernel has **exactly zero DC**. This corrects a slight error: the even/cosine
channel previously carried a small residual DC (~1.5% of the ideal-response
scale at σ=0.5λ — the "~1% DC leak" that was flagged in the 2026-07-11 session
findings). A uniform region of value `c` therefore leaked a spurious response
∝ `c`, inflating its modulus and biasing its phase toward "line".

It was harmless in practice only because EMNIST backgrounds are black
(`c ≈ 0`, so the leak is ≈ 0), which is why it went unnoticed — but it was
still wrong, and any bright uniform region would have produced a phantom
even-channel response. Mean-subtraction makes every uniform region, at any
brightness, give exactly 0. The positive-lobe (ideal-response) normalizer is
re-derived *after* the subtraction, since deepening the negative lobes changes
the positive-lobe sum. Verified: post-fix `|DC| ≈ 1e-7`, positive lobe still
sums to 1.0 at every scale, and per-scale max moduli on the test "T" are
unchanged at 0.15 / 0.35 / 0.63 / 0.47 (λ=3/6/12/24). Baked into the kernel, so
no call site needed touching.

## New: configurable Gabor anisotropy (`GABOR_ASPECT`)

`gabor_kernel` grew an `aspect` keyword (defaulting to the new Config constant
`GABOR_ASPECT`): the ratio of the σ **along** the contour to the σ **across**
it.

- `aspect = 1` → the original isotropic (circular) envelope.
- `aspect > 1` → elongated along the stroke (narrower orientation tuning,
  simple-cell-like), while the across-contour σ — hence the carrier bandwidth /
  scale selectivity — is left untouched.

This replaces the old unused `γ` keyword, whose sense was inverted (γ compressed
the tangent axis) and which never enlarged the kernel support, so an elongated
envelope would have been truncated. `aspect` grows the support with
`ceil(2σ·max(1, aspect))`, so the longer envelope is fully represented; at
`aspect = 1` the support and every kernel value are **identical to before**
(verified: EMNIST "T" per-scale max moduli 0.146 / 0.353 / 0.626 / 0.469 at
aspect 1, matching the recorded values; support 25²→73² and still zero-DC /
lobe-normalized at aspect 3). `aspect` threads through `gabor_lift`; the
T-junction path needs no change since `t_junction_lift` just consumes the Gabor
samples.

All three test notebooks (`Test_CreateGaborLifting`, both
`Test_CreateTJunctionLifting*`) got a **Gabor-aspect slider** (1.0–4.0, default
1.0) wired into their `gabor_lift(...; aspect=…)` call, and the overlay glyphs
reflect the elongation honestly: the filled stroke ellipses scale their
along-contour semi-axis by `aspect` (across-contour unchanged), and in the
T-junction notebooks the dashed receptive-field outline is now an **ellipse**
(λ across, λ·aspect along the stem direction) rather than a radius-λ circle —
both reducing to the originals at aspect 1.

## Changed: T-junction phase-compatibility now fires on T's **and** all 4 corners

`t_junction_lift`'s phase-compatibility term was
`(1 + cos(cross.phase − stem.phase)) / 2`. That matches phases directly, which
is correct for line⟂line T's (even phases, 0/π) but **mishandles corners**:
edge⟂edge phases are ±90°, and that ±90 sign is an artifact of canonicalizing
orientation to [0, π) (flipping a detector's normal conjugates its response,
φ→−φ). So the old term fires on corners whose bright region straddles one
diagonal (phases ≈ 90/90 → Δ≈0) but **cancels** the mirror-image corners on the
other diagonal (≈ 90/270 → Δ≈π) — purely a convention side effect, not a real
distinction. (Note the trap: opposite-polarity line crossings *also* sit at
Δ≈π, so you can't simply symmetrize in Δφ without readmitting those.)

New term:

    phase_compat = clamp(cos φ_s·cos φ_c + |sin φ_s·sin φ_c|, 0, 1)

equivalently `clamp(max(cos(φ_s−φ_c), cos(φ_s+φ_c)), 0, 1)`. It matches the
*sign* of the even (line) part (0 vs π is a real bright/dark polarity) but is
*sign-blind* on the odd (edge) part. Properties (verified on idealized phases):
same-polarity T's (0,0)/(π,π) → 1; all four corners (90,90)/(270,270)/(90,270)/
(270,90) → 1; opposite-polarity crossing (0,π) → 0; line⟂edge mix (0,90) → 0.
It is invariant to both contrast inversion and the per-detector normal-sign
artifact, while still distinguishing same- from opposite-polarity lines. The
old line is kept in a comment with this rationale.

## New: `Test_TJunction_CornerDemo.jl`

Interactive Pluto notebook comparing old vs new compat on controlled synthetic
stimuli — the four corner orientations, a T, an opposite-polarity crossing, and
a straight-edge control. Per-pairing probe confirms the effect on real Gabor
responses: the two anti-diagonal corners come out at φ ≈ (312°, 104°) → old
compat 0.06, new 0.56 (old misses, new keeps), while the T and the other two
corners read even and both terms keep them.

Visualization lesson learned: a plain old-vs-new dot overlay looks nearly
identical, because it is max-pooled over the 8 directions per grid point — at
almost every point *some* direction gives a compatible even/even pair both
terms accept, drowning out the one corner pairing that differs. Replaced the
old panel with a **rescue map**: per grid point, the largest `new − old` gap
over directions. Agreeing points cancel to ~0 and vanish; only junctions the
new term rescues light up (the corner for a corner stimulus; the bar *ends*,
not the junction, for a T — since both terms already handle T's).

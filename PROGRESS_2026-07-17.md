## New: `Dense_Gabors/` — dense sampling instead of a sparse grid

The main pipeline lifts an image into a *sparse grid* of Gabor samples. This
subfolder asks the opposite question: convolve with a **dense, per-pixel**
bank of oriented Gabors and read discrete, typed keypoints — **endpoint /
corner / T-junction / X-crossing** — straight off the resulting oriented-energy
field, before any learned patch dictionary.

The folder now holds the Python sources this is ported from, their design
notes, and the Julia ports:

```
Dense_Gabors/
  gabor_orientation_demo.py            Python: fixed-scale argmax-orientation analysis
  Gabor_Orientation_Demo.jl            Julia port of the above (moved here from the root)
  Gabor_Feature_Layer.jl               Julia port of the feature-type layer  [NEW]
  Gabor_Feature_Layer_MultiScale.jl    Round-2 multi-scale extension          [NEW]
  Gabor_feature_layer_python/
    feature_layer.py                       Python: the feature-type layer
    Gabor_feature_layer_design_notes.md    Rationale for every threshold/choice
```

`gabor_orientation_demo.py` and `Gabor_Orientation_Demo.jl` moved out of the

## New: `Gabor_Feature_Layer.jl` (Julia port of `feature_layer.py`)

Three cheap per-pixel operations read off the oriented-energy stack:

1. **end-stopping** along the winning orientation (segment termination → endpoint);
2. **orientation-profile peak structure**, bimodality-gated (two genuine
   orientation peaks → corner-family);
3. a **multi-radius ring "spoke count"** (number of contour branches → junction order).

The architecture — the design notes' headline lesson — is *propose ≠ classify*:
the two dense operations propose keypoints and make the corner-vs-endpoint
call; the sparse, fragile geometric ring count is only allowed to **refine** an
already-proposed point (corner → T → X), never to propose one.

Faithful function-for-function port (`prep`, `energy_stack`, `end_stopping`,
`peak_maps`, `arc_count`, `classify_point`, `nms`, `detect`), same thresholds,
same config as the Python `__main__` (`n_orient=60`, chars A/K/X/T). Port
substitutions:

| Python | Julia |
|---|---|
| `scipy.signal.fftconvolve(…, "same")` | `imfilter(…, Fill(0))` |
| `ndimage.shift` / `map_coordinates` (`order=1`, constant) | one shared `bilinear` sampler |
| `gaussian_filter1d(σ=1, axis=0, mode="wrap")` | explicit circular 9-tap Gaussian along θ |
| cubic `ndimage.zoom` upsample | bilinear (as in the orientation-demo port) |
| `emnist` package | `LoadEMNIST` |

Verified by running the notebook headless. On real EMNIST samples: **A** → 4
corners (apex + crossbar meets) + 3 endpoints; **K** → 2 **T-junctions** at the
arm/leg meets + 3 corners + 4 endpoints; **T** → corners at the crossbar/stem
meet + endpoints at the crossbar end and stem foot; **X** → the crossing
under-reads as corners (8) + 7 endpoints, which is exactly the single-scale
limitation the design notes predict (the ring resolves only 2 branches at this
glyph/scale).

One display gotcha worth remembering: **GR renders a "transparent" marker fill
as black**, so matplotlib's `facecolors="none"` open markers can't be
reproduced with `scatter!`. The open squares/circles/triangles/stars are drawn
as small line paths instead (`marker_path`).

## New: `Gabor_Feature_Layer_MultiScale.jl` — round 2, and what it actually showed

The design notes' "big one" is that everything above runs at **one** Gabor
scale, and their proposed round-2 fix is: run at ≥ 2 scales and require a
corner-family point to **survive as bimodal across scales** — a true corner
keeps two orientation peaks regardless of receptive-field size, while a curve's
broad rotating peak should collapse to one orientation once the receptive field
drops below its radius of curvature, and drop out.

This notebook implements that: every operation unchanged, now run at **3
scales** (λ = 8/12/18, σₙ/σₜ/`ksize` derived from λ at fixed ratios so the λ=12
bank reproduces the single-scale notebook exactly), with `end_stop_d = λ` and
ring radii `λ·(1.25, 1.5, 1.75, 2.0)` — which give `(15, 18, 21, 24)` at λ=12,
matching the single-scale notebook. At `nS = 1` the whole thing reduces to
`Gabor_Feature_Layer.jl` byte-for-byte (verified against the λ=12 result).

It was validated against real data rather than assumed to work, and that
produced three findings — two of them negative, and kept as such:

### Finding 1 — the corner-vs-curve premise does **not** hold on real handwriting

Measured directly: fraction of on-ridge pixels the bimodality gate calls
corner-family, as λ increases:

```
A (real corners):     0.29  0.12  0.11  0.10  0.08  0.06  0.03      λ = 6 … 22
K (real corners):     0.20  0.11  0.10  0.10  0.12  0.14  0.16
T (real corners):     0.22  0.13  0.12  0.11  0.10  0.10  0.11
X (real junction):    0.29  0.20  0.21  0.19  0.20  0.22  0.29
S (curve, no corner): 0.27  0.21  0.18  0.18  0.19  0.24  0.33
```

At every scale usable before filter noise dominates, **`S` sits in `X`'s range,
not `A`'s** — the mechanism does not separate a curve from a real corner on this
sample. Going finer doesn't rescue it either: on `S` alone the bimodal fraction
*rises* below λ≈6 (0.27 → 0.37 → 0.56 → 0.79 at λ = 6/5/4/3), which is the
kernel shrinking toward the stroke width and reading noise, not the curve
resolving. Fig 6 in the notebook shows the same thing as maps: cornerness along
the curve doesn't meaningfully thin from λ=18 to λ=8.

This is left as an **open problem**, documented in the notebook rather than
papered over. One untested candidate for round 3: a peak-*separation-angle*
discriminator — the angle between the two orientation peaks should stay roughly
constant across scale for a true corner but grow with receptive-field size for a
curve — i.e. use the *geometry* of the bimodality rather than its presence.

### Finding 2 (a real bug, fixed) — a flat cross-scale ring pool over-fires

The obvious way to extend the ring count to 3 scales — pool all 12
(scale × radius) samples and keep the existing "order confirmed at ≥ 2 samples"
rule — is wrong: 2-of-12 agreement is far weaker than the original 2-of-4, and
spurious T/X junctions fired all over `A`, `X`, `T` and mid-stroke on `S`.

Fix: a **two-level vote**. The single-scale rule (order confirmed at ≥ 2 of
that scale's 4 radii) runs independently *per scale*; then the order itself must
agree across **≥ 2 of the 3 scales**. This keeps the agreement strength constant
regardless of scale count and still reduces to `classify_point` at `nS = 1`.
Fig 7 shows it directly: single-scale `S` sprouts a spurious T mid-curve, the
multi-scale vote removes it.

### Finding 3 — that fix has a cost, and no threshold avoids both failure modes

At the shipped ≥2-of-3 threshold, **`K`'s two genuine T-junctions also drop to
"corner"** — they confirm order-3 at λ=12 but not at λ=8 or λ=18, so the ring
radii scaled to those λ over/under-shoot this sample's branch structure.
Loosening to "any one scale confirms" restores `K`'s T's but reopens Finding 2's
false-positive class everywhere else (verified: `A`/`X`/`T`/`S` all sprout
spurious T's again, plus a false `X` on `K` itself).

This isn't a threshold to retune away — it's evidence that **one fixed set of
ring radii-per-scale doesn't resolve junction structure robustly across scales**
on real handwriting. `need = 2` ships as the safer default (no spurious
multi-way junctions, at the cost of some missed real ones).

**Net:** the multi-scale infrastructure works and the ring-vote fix is real; the
specific "bimodality survives across scale" mechanism for corner-vs-curve does
not, on this data.

## New: `New_Gabor_FPE/` — ray harmonics, and why the above is a dead end

**Status: entirely untracked** (nothing in git), produced in a separate Claude
session on 2026-07-15/16, with its own `.claude/settings.local.json`. It was
not visible to `git log` and so was missed in the first draft of this report.
Its own handoff doc, `New_Gabor_FPE_summary.md`, is the primary source; the
summary below is a pointer, not a replacement.

This line of work **supersedes** the `Dense_Gabors` line above. Its §6 lists
`feature_layer.py` as "works, **superseded**", and its §7.4 says explicitly:
*"Do **not** implement the 'bimodality survives across scales' test from the
old design notes — same GOFAI disease."* That is exactly what
`Gabor_Feature_Layer_MultiScale.jl` implements.

### The root-cause diagnosis

The orientation fibre `E(x, y, θ)` is **π-periodic**. Junction type is a **2π
(directional)** property — it is about *which rays leave a point*. Orientation
mod π cannot distinguish "ray going east" from "ray going west", so it cannot
count rays. Measured at the centre of canonical figures (cosine similarity of
`E(θ)`):

```
L-corner   vs T-junction : 0.9031
L-corner   vs X-crossing : 0.8868
T-junction vs X-crossing : 0.9234
```

**L, T and X are effectively the same vector in the orientation fibre** — all
bimodal, peaks at the same orientations. No downstream FPE / attention /
learning can recover what the representation never encoded.

This is the deeper explanation of the two negative findings recorded above,
which were arrived at independently and empirically:

- **Finding 1** (`S`'s bimodal fraction sits in `X`'s range, not `A`'s) is not
  a quirk of one EMNIST sample — the discriminating information is not in the
  representation at any scale.
- **Finding 3** (the ring probe is scale-fragile; no threshold avoids both
  failure modes) is explained by the summary's remark that *"the ring was
  sampling mod-2π structure because the fibre couldn't. The GOFAI was a patch
  over a representational hole, not a stylistic lapse."*

### The fix: ray profile → circular harmonics (= FPE with integer frequencies)

At each point `p`, for `φ ∈ [0, 2π)`:

```
R(p, φ) = E( p + d·u(φ),  θ = φ mod π ),      u(φ) = (cos φ, sin φ)
```

"Is there a contour at distance `d` in direction `φ`, oriented *along* `φ`?"
The `d`-offset is what turns a mod-π quantity back into a mod-2π one (east and
west read different pixels). `R` is 2π-periodic with **one lobe per branch**:
endpoint → 1, straight → 2 opposite, L → 2 adjacent, T → 3, X → 4. Its Fourier
coefficients `cₙ` are the type signature:

| config | c₀ | \|c₁\|/c₀ | \|c₂\|/c₀ |
|---|---|---|---|
| endpoint (1 ray) | 1 | 1.000 | 1.000 |
| straight (2 opposite) | 2 | 0.000 | 1.000 |
| L-corner (2 @ 90°) | 2 | 0.707 | 0.000 |
| **T-junction (3 rays)** | **3** | **0.333** | **0.333** |
| X-crossing (4 rays) | 4 | 0.000 | 0.000 |

`c₀` ≈ ray count; `|c₁|/c₀` ≈ asymmetry (1 = endpoint, 0 = centrally
symmetric). `(c₀, |c₁|/c₀)` alone separates all five. Junctions are simply the
**brightest points of the `c₀` map** — that is the detector, no ring probe.
Note `c₄` is degenerate on the canonical set (all rays on the 90° grid); the
discrimination lives in `c₀, c₁, c₂`.

**Why this is not GOFAI:** for fixed `φ`, the term is one orientation channel
*rigidly shifted*. The whole expression is "K rigid shifts, weighted by
`e^{-inφ}`, summed" — a linear filter over the lifted `(x, y, θ)` field, same
operation class as the Gabor convolution that produced `E`. No counting, no
thresholds, no run-length analysis, no branching. `|cₙ|` is rotation-invariant
by construction, and rotation acts as binding (`V → z^α ⊙ V`).

**FPE *is* the harmonic expansion** (`V = Σ_φ R(φ)·z^φ`) — so this is not a new
module bolted on; it is the FPE layer with the base fixed.

### Two real bugs it diagnosed in the old VSA notebook

- **Bug A — `Θ_BASE` is not periodic.** Continuous base phases make `z^θ`
  non-π-periodic, so `θ=0` and `θ=π` encode to maximally different vectors.
  The `2*` in the exponent cannot fix this: periodicity is a property of the
  base **frequencies**, not the exponent. Requires `φₖ ∈ ℤ`. (Check:
  `|⟨z⁰, z²ᵖⁱ⟩|/d` must be 1.0.)
- **Bug B — `|⟨M, C ⊙ M⟩|²` is a bag of pairs.** It pools over the whole image,
  counting relative configurations without binding them to a location — a
  global second-order statistic, the Portilla–Simoncelli texture family, known
  *not* to represent shape. Junction type is a conjunction of relations **at one
  point**; global pooling destroys precisely that.

Hence the forced architectural split: **dense/convolutional with no bundling**
(Gabor → `E` → `cₙ` maps), then **sparse keypoint selection** (top-N of `c₀`),
then **FPE bundling** of those keypoints bound to object-relative polar position
`(r, α)`. Bundling `N` items into `d` dims has SNR ≈ `√(d/N)`, so bundling
~600k dense tokens into `d=1024` is pure noise; a few dozen keypoints is fine.
(Mirrors dense retinotopic V1/V2/V4 maps vs sparse object codes in IT; `(r, α)`
is the Pasupathy–Connor angular-position code.)

### Design rules from that work (do not violate without a reason)

- **No thresholds, no counting, no `if/elif` on feature type.**
  `(c₀, |c₁|/c₀, |c₂|/c₀, …)` **is** the descriptor — bind the continuous
  vector and let similarity do the work.
- **Do not classify into symbolic labels.** A 30° corner has signature
  ≈(0.97, 0.87), close to an endpoint — and that is *correct*: a sharp corner
  really is nearly a doubled-back single ray. Corner-ness is graded, as V4
  curvature tuning is graded. The discrete labels were the GOFAI residue.
- **Never bundle globally before a locality-dependent readout** (Bug B).
- **Any circular FPE variable needs integer base frequencies** (Bug A).
- **Orientation mod π ⇒ encode 2θ. Ray direction mod 2π ⇒ encode φ.**

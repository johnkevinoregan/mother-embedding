# mother-embedding — progress summary, 2026-07-08

## Goal

Rewrite the monolithic Pluto notebook `~/Julia/DATABASES/FPE_emnist_Gabor-pair_multiletter.jl`
(a VSA/Fractional-Power-Encoding letter classifier built on a Gabor-filter
front end) as a set of independent, individually testable Julia modules,
each `include`-able on its own (`include("X.jl"); using .X`), paired with
its own small Pluto notebook for visually sanity-checking that component
before moving to the next.

- Location: `/home/kevin/claude-code/mother-embedding`, a self-contained

  `run_pluto.sh`) — a browser refresh alone does not pick up on-disk
  changes reliably.

## Components completed

### `Config.jl`
Single source of truth for project-wide constants, `include`d + `using`'d
before any component module. Currently holds the Gabor-lifting constants:
`IMG_SIZE`, `NATIVE_SIZE`, `SCALES`, `ORIENTATIONS`, `OVERLAP_FRAC`.
Referenced **bare** everywhere (`SCALES`, not `Config.SCALES`) — modules
and notebooks alike just `using .Config`. To change a value: edit
`Config.jl`, restart the Pluto server (these are `const`s — no live
mutation).

### `LoadEMNIST.jl` + `Test_LoadEMNIST.jl`
Parses the EMNIST balanced-train IDX files and buckets images by class.
- `load_emnist(; data_dir, n_images_to_load=20000, n_classes=5,
  class_order=default_class_order())` → named tuple `(images, labels,
  selected_classes, class_names, class_images)`.
- `to_image(m)` — fixes a display-orientation bug found during testing: raw
  EMNIST samples are already correctly oriented; the *display* needs a row
  flip (`reverse(dims=1)`) to render upright in `Plots.heatmap`, which
  defaults to bottom-up. (The original source notebook's `rotr90` approach
  was actually wrong for this purpose.)
- Test notebook: sliders for class count / images scanned, per-class image
  counts, a 5-image sample grid, and a per-class count bar chart.

### `CreateGaborLifting.jl` + `Test_CreateGaborLifting.jl`
Convolves an image with a bank of complex Gabor filters and samples the
response on a grid.
- `GaborSample`: `(x, y, θ, s, modulus, phase)` — carries the full complex
  response (modulus **and** phase), not just magnitude as in the original.
- `gabor_kernel(λ, θ)` — complex Gabor kernel, energy-normalized.
- `imresize_simple` — bilinear upscale 28×28 → 56×56 (kept as in the
  original; bicubic via ImageTransformations was considered but not used).
- `gabor_lift(image; scales, orientations, overlap_frac, img_size)` — raw,
  **unthresholded** output: every grid point at every scale/orientation is
  returned (the original applied a response threshold and dropped weak
  samples; this version doesn't). The only normalization: each modulus is
  divided by the kernel wavelength λ, so moduli are comparable across scales
  (added 2026-07-09 — see resolved issue 2 below). Boundary handling uses
  `Pad(:replicate)` (kept as in the original) — noted that at the largest
  scale the kernel radius is a large fraction of the image size, so most
  border tokens are influenced by replicated edge pixels.
- Simplified from the original's two-pass (separate real/imag `imfilter`)
  approach to a single complex-kernel `imfilter` call — verified numerically
  identical (~1e-7, float roundoff) and roughly halves the convolution work.
- `compute_response_grid` kept, commented out, for possible future use
  (superseded for normal use since `gabor_lift` itself now returns every
  grid point unthresholded).
- Test notebook: single-filter overlay (phase shown as line hue, modulus as
  line opacity), a modulus histogram, and a full scale×orientation panel
  grid for one image at a glance. Orientation-matched filters do light up
  along matching strokes, confirming the *orientation* logic — but see the
  `.x`/`.y` bug noted under "Known issues" below, found the same day: the
  *position* labeling was backwards until fixed, so earlier "looks aligned"
  visual checks in this notebook were less rigorous than they should have
  been (worth a fresh look next session).

### `CreateTJunctionLifting.jl` + `Test_CreateTJunctionLifting.jl`
Not part of the original notebook's pipeline — a new component detecting
T-junction candidates (a line segment coupled with an orthogonal segment
displaced along the first segment's own direction) directly from
`CreateGaborLifting`'s output, without going through FPE encoding.
- `TJunctionSample`: `(x, y, α, s, strength)`. `α` is **2π-periodic** (8
  possible values), unlike Gabor orientation (π-periodic) — needed to
  distinguish e.g. a stem pointing left from one pointing right, even though
  both share the same underlying Gabor orientation.
- `t_junction_lift(gabor_samples; scales, orientations, overlap_frac,
  img_size)` — for every sampled grid point P and each of its up to 8 grid
  neighbors N: P is a candidate stem (Gabor orientation = direction P→N, mod
  π), N is a candidate crossbar (orthogonal, mod π). Each lookup snaps to
  whichever available orientation is closest (so this works with any
  orientation count/spacing, not just a set closed under 90° rotation, per
  request). Strength = `min(modulus_P, modulus_N) × phase_compatibility`,
  where `phase_compatibility = (1 + cos(phase_N - phase_P)) / 2` — high when
  the stem and crossbar "look the same" (same line/edge polarity), low when
  they don't. No thresholding, matching `gabor_lift`'s philosophy.
- Deliberately untyped (`gabor_samples::AbstractVector`, no dependency on
  `CreateGaborLifting.jl`): an early version included that module directly
  and broke, because two separate `include`s of a module defining a struct
  type create two distinct, incompatible types — `t_junction_lift` only
  needs `.x .y .θ .s .modulus .phase` fields, so it doesn't need the type at
  all.
- Test notebook: top-N strongest candidates and a best-per-location
  scale×panel view, both drawn as literal T glyphs (stem + perpendicular
  crossbar, each of length `scale/3`) with color saturation proportional to
  strength.
- Status: **works correctly at the smallest scale, but the combined
  cross-scale "top N" view is currently misleading** — see "Known issues"
  below. Do not treat today's earlier claims of "verified, clusters at the
  junction" as settled; they were based on a since-fixed position bug and
  were re-examined at the end of the day with more rigorous, scale-separated
  evidence.

## Known issues to resolve next session

1. **Found and fixed today: `.x`/`.y` were transposed in `gabor_lift`.**
   Internally it loops `for cx in ...rows...; for cy in ...cols...;
   response[cx, cy]`, and the original code put the row value into `.x` and
   the column value into `.y` — backwards from the normal `x=horizontal,
   y=vertical` convention. This silently mis-positioned every overlay in
   both `CreateGaborLifting` and `CreateTJunctionLifting`'s test notebooks
   (though the underlying detection math itself was unaffected, since it
   works on raw grid indices, not the `.x`/`.y` labels). Fixed by swapping
   which loop variable feeds `.x` vs `.y` in `CreateGaborLifting.jl`;
   verified with synthetic images at known pixel locations, both before and
   after the fix.

2. **RESOLVED 2026-07-09: `TJunctionSample.strength` wasn't comparable
   across scales.** Larger Gabor kernels integrate more image energy even
   when energy-normalized (measured on a real EMNIST "T": max modulus 1.20 /
   3.85 / 5.82 at λ=6/12/24 — roughly proportional to λ), so a combined "top
   N across all scales" ranking was dominated by large-scale matches (top-30
   scale composition was 0/16/14). Fixed at the source: `gabor_lift` now
   divides each modulus by λ. After the fix, per-scale max moduli are 0.20 /
   0.32 / 0.24, top-30 composition is 8/16/6, and the single strongest
   T-junction candidate on the test "T" is the true junction (stem grid
   point one step below the crossbar, α pointing up at it).
